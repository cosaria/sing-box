## Affected API Endpoints
- POST /api/inbounds — add inbound configuration
- GET /api/inbounds — list all inbounds
- GET /api/inbounds/:id — view single inbound
- PUT /api/inbounds/:id — modify inbound
- DELETE /api/inbounds/:id — delete inbound
- POST /api/server/start|stop|restart — service control
- GET /api/server/status — engine status
- GET /api/stats — traffic statistics
- GET /sub/:token — subscription endpoint (public, no auth)
- GET /api/system/info — system information
- POST /api/system/update — trigger self-update

## Key Interactions to Verify
- Add Shadowsocks inbound → config saved to SQLite → JSON generated → engine reloaded → share link returned
- Add VLESS-REALITY inbound → correct JSON with reality fields
- Add Trojan inbound → correct JSON with Trojan fields
- Auth: requests to /api/* without token → 401
- Auth: requests to /sub/:token → no auth required, works
- Engine lifecycle: start → status=running → add inbound → reload → still running → stop

## Edge Cases
- Add inbound with port already in use
- Delete last inbound → engine has no inbounds
- Invalid protocol parameters (bad method, out-of-range port)
- API request with malformed JSON body
- Subscription token: invalid token → 404 (no info leak)
- Stats query with no traffic data → empty array, not error
- Self-update when binary is running as systemd service

## Critical Paths
- Full lifecycle: add config → engine start → verify proxy works → reload → verify still works → stop
- SQLite → JSON generation → engine reload (single source of truth chain)
- Crash recovery: engine panics → recovers → restarts within same process
