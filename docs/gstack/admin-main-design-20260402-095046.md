## Problem Statement

现有的 sing-box 一键管理脚本是纯 Bash 编写的（~965 行），依赖 wget、jq、qrencode 等外部工具。虽然功能完整且架构清晰（刚完成模块化重构），但 Bash 带来了固有局限：运行时依赖多、无类型安全、错误处理脆弱、无法实现进程内监控。

用户希望用 Go 重写为单二进制分发的 TUI 工具，并嵌入 sing-box 核心作为库而非管理外部进程。

## What Makes This Cool

**核心亮点：单二进制 = 代理 + 管理器。**

这不是"把 Bash 翻译成 Go"。嵌入 sing-box 作为 Go 库意味着：

1. **零依赖部署。** 一个文件丢到服务器就能用。不需要 bash、wget、jq、qrencode。
2. **热重载。** 改配置后 `box.Close()` + `box.New()`，亚秒级生效（已有连接会短暂断开，客户端自动重连）。
3. **内部状态监控。** 连接数、流量统计、per-inbound 状态，通过 Go 接口直接获取。
4. **零版本不匹配。** 管理器和代理核心永远是同一个版本。
5. **真正的 TUI。** Bubbletea 提供的交互体验是 Bash echo/read 永远达不到的。

市面上所有 sing-box 服务器管理工具都是 Bash 脚本，所有编译型工具都是客户端/桌面 GUI。**编译型 TUI 服务器管理工具是一个空白。**

## Constraints

- **目标平台：** Linux amd64 + arm64（Ubuntu/Debian/CentOS/Alpine）
- **不是客户端。** 用户通过 SSH 连接服务器使用。
- **init 系统：** 必须支持 systemd 和 OpenRC。
- **第一版功能范围：** 现有 Bash 版本的等价功能（安装、配置 CRUD、服务管理、协议支持）。嵌入架构自然带来的能力（热重载、内部状态查询）属于架构副产品，不额外增加面向用户的新功能（如多节点管理、插件系统）。
- **语言：** Go。sing-box 是 Go 写的，嵌入需要同语言。
- **用户界面语言：** 中文。

## Premises

1. 核心目标是服务器端管理工具，跨平台指 Linux 发行版，不是 macOS/Windows。
2. Go 比 Rust 更适合，因为 sing-box 是 Go 写的，可以直接嵌入作为库。
3. 第一版 = Bash 版本等价功能 + 嵌入架构的自然副产品（热重载、状态查询），不在 v1 加入多节点管理或插件系统。
4. Bash 版本已经够用。重写驱动力是工艺追求，不是 Bash 有不可解决的问题。

## Cross-Model Perspective

Claude 子代理独立分析后提出的关键见解：

- **嵌入式架构是最大的差异化。** 所有现有工具都是管理外部 sing-box 进程。嵌入 sing-box 作为库是唯一能产生"Bash 不可能实现"的能力的路径。
- **周末原型策略：** 最高风险项（sing-box-as-library 可行性）应该第一天验证。先 `import "github.com/sagernet/sing-box"` 调用 `box.New()` 跑通 Shadowsocks 流量。如果这步失败，其他方案仍然可行。
- **50% 免费：** Bubbletea 生态（lipgloss、bubbles 组件库）覆盖了所有 UI 需求。sing-box Go 库提供配置结构体和生命周期管理。剩下 50% 是服务管理、协议向导、分享链接、自安装。
- **API 稳定性风险：** `github.com/sagernet/sing-box` 不是稳定公开 API。每次 sing-box 升级可能 break 依赖。需要 go.mod 锁定版本并谨慎升级。

## Approaches Considered

### Approach A: 嵌入式架构（Go + sing-box 作为库） [CHOSEN]

**Summary:** 单二进制同时是代理引擎和管理器。`import "github.com/sagernet/sing-box"` 在进程内运行代理。

**Effort:** L
**Risk:** Medium（API 稳定性）
**Pros:**
- 零依赖，零版本不匹配
- 热重载，内部监控，亚秒级配置变更
- 真正的架构跳跃，不只是"换语言"

**Cons:**
- 二进制体积较大（~15MB+ with sing-box embedded）
- sing-box 内部 API 非公开稳定接口
- 需要深入理解 sing-box 源码

**Reuses:** sing-box Go library, Bubbletea TUI framework

### Approach B: 管理器架构（Go + 外部 sing-box 二进制）

**Summary:** 像 Bash 版本一样管理外部 sing-box 进程，但用 Go 编写。

**Effort:** M
**Risk:** Low
**Pros:**
- 简单，风险低，与 Bash 版本逻辑一一对应
- 更小的二进制（~5MB），sing-box 版本解耦
- 不依赖 sing-box 内部 API

**Cons:**
- 本质上是"换语言"，没有架构上的飞跃
- 仍然需要下载/管理外部 sing-box 二进制
- 无法热重载、无法内部监控

**Reuses:** Bubbletea TUI framework

### Approach C: Rust + Ratatui

**Summary:** Rust 编写，调用外部 sing-box 二进制。

**Effort:** M-L
**Risk:** Medium（无法嵌入 sing-box）
**Pros:**
- 最小二进制（~2-3MB），内存安全
- Ratatui 生态成熟

**Cons:**
- 无法利用 sing-box Go 库，只能管理外部进程
- Rust 学习曲线可能较陡
- 跨语言调用 sing-box 不实际

**Reuses:** Ratatui TUI framework

## Recommended Approach

**方案 A: 嵌入式架构。**

理由：这是唯一一个让重写"值得"的方案。方案 B 和 C 本质上是"用更好的语言做同样的事"。方案 A 让单二进制同时承担代理和管理两个角色，解锁了 Bash 不可能实现的能力（热重载、内部监控、零版本不匹配）。

风险缓解：第一步验证 `github.com/sagernet/sing-box` 的 `box.New()` API 是否可用。如果 API 不稳定或不可用，可以降级到方案 B。

## Engine Lifecycle

嵌入式 sing-box 引擎有四个状态：

```
Unconfigured → Configured → Running ←→ Reloading
                                ↑ (success)    │ (failure)
                                └── Configured ←┘
```

- **Unconfigured:** 没有配置文件。可以通过 TUI/CLI 添加配置。
- **Configured:** 有配置文件但引擎未运行。`sing-box serve` 从此状态自动过渡到 Running。
- **Running:** 引擎在运行，代理流量中。
- **Reloading:** 配置变更触发。`box.Close()` → `box.New()` → `box.Start()`。成功则回到 Running，失败则回退到 Configured 并报错。**注意：** 重载期间已有活动连接会被断开（通常 <1 秒），客户端需自行重连。

### Runtime Modes

二进制有两种运行模式，通过 cobra 子命令区分：

```
sing-box                          # 无参数 → 启动 TUI 交互菜单
sing-box serve                    # 守护进程模式（systemd/openrc 调用）
sing-box add [protocol] [args]    # CLI 添加配置
sing-box del [name]               # CLI 删除配置
sing-box info [name]              # CLI 查看配置
sing-box update [core|script]     # 自更新
sing-box status                   # 查看服务状态
sing-box start|stop|restart       # 服务控制
```

`sing-box bin` 子命令不再需要（因为 sing-box 核心已嵌入），予以移除。用户如需原始 sing-box CLI 功能，可通过 `sing-box generate` 等直接暴露的子命令访问。

### Management-to-Daemon Communication

管理命令需要与守护进程通信以触发配置重载：

1. **PID 文件：** 守护进程启动时写 `/run/sing-box.pid`。
2. **检测守护进程：** 管理命令先检查 PID 文件是否存在且进程存活。
3. **重载通知：** 向守护进程发送 `SIGHUP`，守护进程收到后重新扫描 `$CONF_DIR` 并重载。
4. **守护进程不在运行时：** 管理命令仅写配置文件，跳过重载通知。下次 `sing-box serve` 启动时自动加载新配置。

## Logging

当 sing-box 嵌入为库运行时，日志通过 sing-box 的 `option.LogOptions` 配置：
- **守护进程模式 (`serve`):** 日志写入 `$LOG_DIR/access.log`（与 Bash 版本一致）
- **TUI 模式:** 日志写入文件，TUI 的"查看日志"功能 tail 该文件
- sing-box 内部 logger 输出到 Go `io.Writer`，可以同时写文件和内存 ring buffer（供 TUI 实时显示）

## Config Storage

保持与 Bash 版本兼容的存储模型：
- 每个协议配置一个独立 JSON 文件在 `$CONF_DIR/`（如 `Shadowsocks-12345.json`）
- `$CONFIG_JSON` 保存 log/dns/outbound 全局设置
- Go 版本启动时扫描 `$CONF_DIR/*.json`，解析所有 inbound，合并到一个 `option.Options` 传给 `box.New()`
- 这样现有 Bash 版本的配置文件无需任何修改即可被 Go 版本读取

## Self-Update

自更新使用原子替换模式：
1. 从 GitHub Releases 下载新版本到 `/usr/local/bin/.sing-box.tmp`（必须与目标在同一文件系统，避免 `os.Rename()` 跨文件系统 `EXDEV` 错误）
2. 校验 checksum
3. `os.Rename(".sing-box.tmp", "sing-box")` 原子替换
4. 通过 systemd `restart` 重启守护进程（新二进制生效）

## Open Questions

1. **sing-box Go API 稳定性。** `box.New()` / `box.Start()` / `box.Close()` 接口在跨版本升级时是否保持兼容？需要读 sing-box 源码确认。M0 原型阶段验证。
2. **Build Tags 与 CGO。** sing-box 使用 build tags 控制功能（`with_quic`, `with_grpc`, `with_utls`, `with_reality_server` 等）。Shadowsocks 最小构建可能不需要额外 tags，但完整协议支持需确定哪些 tags 必须启用。部分 tags 可能引入 CGO 依赖，影响 linux/arm64 交叉编译。M0 阶段确认纯 Go 构建是否可行。
3. **二进制体积。** 嵌入 sing-box 后二进制可能 15-20MB（取决于 build tags）。对服务器场景可接受（sing-box 官方二进制本身约 20MB），但应精简不需要的 tags。

**已解决的问题：**
- ~~自更新机制~~ → 见 Self-Update 章节（原子替换模式）

- ~~配置迁移~~ → v1 不做自动迁移，配置文件格式与 Bash 版本兼容，无需修改

### Protocol 接口

Bash 版本的 `protocol_ask()` 是阻塞式交互输入，但 Bubbletea 是事件驱动的 Elm 架构（Init/Update/View），不能在 Update() 里阻塞等待输入。因此 Protocol 接口将 **数据关注点** 和 **UI 关注点** 分离：协议声明自己需要哪些字段和校验规则，TUI wizard 负责驱动收集过程。

```go
// Config 是协议配置的通用容器
type Config struct {
    Protocol string            // 协议类型标识（如 "shadowsocks"）
    Port     int
    Fields   map[string]string // 协议特定字段（password, method 等）
    FileName string            // 配置文件名（如 "Shadowsocks-12345.json"）
}

// FieldSpec 描述一个可收集/可编辑的字段
type FieldSpec struct {
    Key          string   // 字段名（如 "method"）
    Label        string   // 显示名（如 "加密方式"）
    Type         string   // "text" | "select" | "port"
    Default      func() string          // 默认值生成器
    Options      []string               // Type="select" 时的选项列表
    Validate     func(string) error     // 校验函数
}

type Protocol interface {
    Name() string                           // 协议名（如 "Shadowsocks"）
    Fields() []FieldSpec                    // 声明需要收集的字段和校验规则
    JSON(cfg *Config) ([]byte, error)       // 生成 sing-box inbound JSON
    URL(cfg *Config, addr string) string    // 生成分享链接
    Info(cfg *Config, addr string) string   // 生成展示信息
    Parse(json []byte) (*Config, error)     // 从 JSON 解析回 Config
}
```

**FieldSpec.Type 语义：**

| Type | UI 组件 | 校验规则 |
|------|---------|----------|
| `"text"` | bubbles/textinput 文本框 | 非空，可选自定义 Validate 函数 |
| `"select"` | bubbles/list 选择列表 | 必须从 Options 中选择 |
| `"port"` | bubbles/textinput 数字输入 | 1-65535 范围，自动检查端口占用，Default 生成随机可用端口 |

TUI wizard 读取 `Fields()` 返回的字段规格，为每个字段渲染合适的输入组件，收集完成后构造 `Config` 传给 `JSON()` 生成配置文件。

### 开发顺序

1. **验证 sing-box 嵌入可行性。** `go mod init` → import sing-box → `box.New()` → 跑通 Shadowsocks 流量。这是最高风险项，必须第一步验证。**注意：** sing-box 使用 build tags（`with_quic`, `with_grpc`, `with_utls` 等）控制功能编译。原型需要确定 Shadowsocks 最小所需的 build tags，以及完整协议支持所需的 tags。二进制体积取决于启用了哪些 tags。
2. **平台检测 + 服务管理。** `internal/platform/` + `internal/service/`。systemd/openrc unit 生成和服务操作。
3. **协议框架 + Shadowsocks。** `internal/protocol/` 接口和 Shadowsocks 实现。
4. **TUI。** Bubbletea 主菜单 → 添加配置向导 → 查看/修改/删除 → 服务管理。
5. **分享链接 + QR。** URL 编码和终端 QR 渲染。
6. **安装/更新机制。** install.sh + 自更新。
7. **更多协议。** 参照 Bash 版本原始代码（`c9c33f9` commit）逐个实现。

## Success Criteria

- 单个 `go build` 产出一个可执行文件，丢到 Linux 服务器上就能用
- 功能等价于当前 Bash 版本：安装、配置 CRUD、服务管理、分享链接、QR 码
- Shadowsocks 协议完整支持
- systemd 和 OpenRC 双支持
- 双模式运行：`sing-box serve`（守护进程）和 `sing-box`（TUI 管理）/ `sing-box add|info|del`（CLI 管理）
- 二进制 < 25MB（with sing-box embedded）
- `go test` 覆盖核心逻辑

## Distribution Plan

- **GitHub Releases：** 每个版本发布 `linux-amd64` 和 `linux-arm64` 二进制
- **install.sh：** 一行命令安装：`curl -sSL install-url | bash`，检测架构并下载对应二进制到 `/usr/local/bin/sing-box`
- **CI/CD：** GitHub Actions，tag push 触发 `goreleaser` 构建并发布
- **自更新：** `sing-box update` 从 GitHub Releases 下载最新版本替换自己

## Next Steps

1. **M0 — 可行性验证：** 新 Go module，`import "github.com/sagernet/sing-box"` + `box.New()` 跑通 Shadowsocks 流量。确定所需 build tags。如果失败，降级到方案 B。
2. **M1 — 项目骨架：** cmd/internal 目录结构，platform 检测，service 管理（systemd + openrc）。
3. **M2 — 核心功能：** 协议框架 + Shadowsocks 实现 + TUI 主菜单和配置向导。
4. **M3 — 完整体验：** 分享链接、QR、install.sh、自更新。完成 v0.1 可用版本。
